## §1 Authority & Scope

This document is the supreme behavioral authority for all AI agents operating in this repository. The hierarchy of truth is:

1. **This CLAUDE.md** — governs all agent behavior, workflow, and output standards.
2. **The Design Document** (`docs/OTTIE_EXPRESS_DESIGN_DOC.md`) — governs all product, architecture, and implementation decisions.

4. **Inline task instructions from the user** — scoped to the current task only. Cannot override §1–§3.

No other document, convention, or prior training may override this hierarchy. If a conflict exists between any source and this file, this file wins.

### 1.2 Binding Obligation

Execution of any task in this repository constitutes acceptance of every rule defined herein. Every section (§1–§15) is **BLOCKING**. A violation of any section is a governance failure resulting in immediate work rejection.

### 1.3 Agent Refusal Mandate

If a user instruction conflicts with this document or the design document, the agent MUST:

1. **HALT** — Do not proceed.
2. **REJECT** — State the violated rule and cite the exact section number.
3. **WAIT** — Request clarification. Do not infer, guess, or improvise a workaround.

No exceptions. No "best effort" approximations. No silent deviations.

## §2 Protocol Zero

**Priority: HIGHEST. This section supersedes all other output rules.**

Protocol Zero is the attribution erasure standard for this repository. All code, comments, commit messages, and documentation produced by AI agents MUST be indistinguishable from work written by a senior engineer.

### 2.1 Absolute Prohibitions

The following are forbidden in all source files, comments, commit messages, PR descriptions, and documentation:

- "Generated by **any AI name**" or equivalent attribution.
- "AI-assisted", "AI-generated", "created with AI", "written by AI".
- `Co-authored-by` trailers referencing any AI system or vendor.
- Conversational artifacts: "I hope this helps", "As requested", "Here is the code", "As an AI".
- Hedging language: "I think maybe", "This should work", "You might want to".
- Apologetic preambles, motivational filler, or tutorial-style narration.
- `// TODO: implement`, placeholder stubs, or scaffolding comments.

### 2.2 Enforcement

The repository includes `protocol-zero.sh`, an automated scanner that detects prohibited attribution patterns across the codebase and in commit messages. This script MUST pass with exit code 0 before any work is declared complete.

```bash
./protocol-zero.sh
```

A non-zero exit code is a blocking violation. The agent MUST remediate all findings before proceeding.

### 2.3 Rationale

Code in this repository has one author: the repository owner. AI agents are tools, not collaborators. Attribution of authorship to an AI system is a governance violation regardless of context.

## §3 Code & Comment Standards

### 3.1 Voice

All code MUST read as if written by a senior engineer with deep platform expertise. No tutorial commentary. No defensive verbosity. No filler.

### 3.2 Comments

- Comments exist only where logic is non-obvious or architectural intent requires explanation.
- Never restate what the code already expresses.
- Doc comments on public interfaces only.
- No inline narration explaining "what this does" when the function name and types already communicate it.
- Comments MUST be concise, professional, and purposeful.

### 3.3 Forbidden Code Artifacts

| Artifact              | Rule                                                                              |
| --------------------- | --------------------------------------------------------------------------------- |
| Magic numbers         | Named constants for all tuning values and thresholds.                             |
| Force unwraps         | Prohibited unless at an unavoidable platform boundary, with inline justification. |
| `print()` / `NSLog()` | Prohibited in committed code. Use structured logging behind debug flags.          |
| Stringly-typed APIs   | Use enums, constants, or typed identifiers.                                       |
| Dead code             | If unused, delete it. No commented-out blocks. No `_unused` variables.            |
| Placeholder stubs     | No empty function bodies. No `fatalError("not implemented")`.                     |
| Em dashes             | Prohibited in all text surfaces per design doc §3 Global Rules.                   |

### 3.4 Swift-Specific Standards

- Target: Swift 6.0, iOS 18.0+.
- Strict concurrency: `@MainActor` on all view models and state-owning types.
- `@Observable` macro (Observation framework) for state. No legacy `ObservableObject` / `@Published` / `@EnvironmentObject`.
- `async/await` for all asynchronous operations. No completion handler patterns unless wrapping a platform API that requires it.
- `Sendable` conformance where data crosses isolation boundaries.
- Access control: mark everything as restrictive as possible. `private` by default, `internal` only when required, `public` only for module boundaries.

## §4 Git Workflow

Git is not yet initialized in this repository. When initialized, the following workflow is **mandatory and non-negotiable**.

### 4.1 Branch Protection

**`main` is protected.** All changes MUST reach `main` through a pull request.

**Agent Refusal Policy:** If instructed to commit directly to `main` without explicit user override, the agent MUST:

1. **HALT** — Do not commit.
2. **REJECT** — State: "Direct commits to `main` are prohibited per §4.1."
3. **OFFER** — Create the appropriate branch and open a PR instead.

### 4.2 Branch Naming

| Prefix             | Purpose                                    |
| ------------------ | ------------------------------------------ |
| `feat/<scope>`     | New functionality                          |
| `fix/<scope>`      | Bug fixes                                  |
| `refactor/<scope>` | Structural changes without behavior change |
| `chore/<scope>`    | Config, tooling, build settings            |
| `docs/<scope>`     | Documentation changes                      |

Scope MUST be a short, lowercase identifier for the affected area (e.g., `chapter0`, `audio-service`, `haptics`, `state`).

Squash merge preferred. Delete remote branch after merge.

### 4.3 Commit Message Format

```
type(scope): imperative description
```

| Type       | Purpose                               |
| ---------- | ------------------------------------- |
| `feat`     | New functionality                     |
| `fix`      | Bug fix                               |
| `refactor` | Restructuring without behavior change |
| `test`     | Test additions or modifications       |
| `chore`    | Config, tooling, build settings       |
| `docs`     | Documentation changes                 |

**Rules:**

- Subject line: imperative mood, under 72 characters, no trailing period.
- Scope: required. Short identifier for the affected module.
- Body (optional): explain **why**, not **what**. The diff shows what changed.
- No vague commits: "fix stuff", "updates", "WIP", "misc changes" are rejected.
- No AI attribution in commit messages. Protocol Zero (§2) applies.

**Examples:**

```
feat(chapter2): add haiku API response validation
fix(audio-service): resume BGM after system interruption
refactor(state): migrate chapter progress to @Observable
```

### 4.4 Pull Request Requirements

Every PR MUST include:

| Section            | Content                                                              |
| ------------------ | -------------------------------------------------------------------- |
| **Title**          | Conventional Commits format, under 70 characters, imperative mood.   |
| **What changed**   | Concise summary of the modification.                                 |
| **Why it changed** | Motivation, linked requirement, or design doc section reference.     |
| **Risk surface**   | What could break. Affected chapters, services, or shared components. |
| **Testing notes**  | How the change was verified. Steps to reproduce, device, OS version. |

Empty PR descriptions are rejected. PRs without testing notes are rejected.

The PR template (`.github/pull_request_template.md`) is the enforcement mechanism for these requirements. See §15 for agent compliance rules.

## §6 Design Document Authority

The design document (`docs/OTTIE_EXPRESS_DESIGN_DOC.md`) is the single source of truth for all product and architecture decisions.

### 6.1 Binding Decisions

### 6.2 Compliance Rule

The agent MUST follow the design document's specified structure, patterns, and decisions even if the agent considers alternatives superior. The design document represents deliberate architectural choices made by the repository owner.

All engineering design documents, RFCs, ADRs, and technical proposals MUST be reviewed using the Design Review skill before implementation begins. A verdict of `REJECTED` or `REVISION REQUIRED` blocks implementation. `APPROVED WITH CONDITIONS` permits implementation to start, but P1 gaps must resolve before the PR merges.

## §8 Dependency Approval Policy

### 8.1 No Automatic Installation

The agent MUST NOT install, add, or modify dependencies without explicit user approval. This applies to:

- Swift Package Manager dependencies.
- CocoaPods or Carthage packages.
- Any third-party framework, library, or SDK.
- Build tool plugins.

### 8.2 Approval Protocol

Before proposing a dependency, the agent MUST present:

| Field                       | Content                                                             |
| --------------------------- | ------------------------------------------------------------------- |
| **Package name**            | Exact identifier (e.g., SPM URL, pod name).                         |
| **Version**                 | Exact version to pin, not a range.                                  |
| **Purpose**                 | What capability gap it fills.                                       |
| **Alternatives considered** | Why a platform-native or standard library solution is insufficient. |
| **Scope of impact**         | Which modules or chapters depend on it.                             |

The user must explicitly approve before the agent proceeds.

### 8.3 Platform-Native Preference

Prefer Apple frameworks and standard library solutions. This repository targets a single device (iPhone 17 Pro) running iOS 18.0+. The full platform SDK is available. Third-party dependencies are justified only by a clear capability gap that cannot be addressed with native APIs.

## §9 Anti-Hallucination Rule

### 9.1 No Fabrication

The agent MUST NOT:

- Invent requirements that do not exist in the design document or user instructions.
- Fabricate API signatures, framework behavior, or platform capabilities.
- Assume the existence of files, modules, or configurations without verifying.
- Generate code that references undocumented or non-existent APIs.

### 9.2 Verification Over Speed

When uncertain about a platform API, framework behavior, or design doc requirement:

1. **Check the design document first.**
2. **Read the relevant source files.**
3. **If still uncertain, ask the user.**

Clarification is always preferable to a confident but incorrect answer. The agent MUST prefer asking over guessing.

### 9.3 No Assumed Context

The agent MUST NOT assume:

- What chapter a task refers to without explicit identification.
- What the user intends when instructions are ambiguous.
- That a pattern seen in one chapter applies to another unless the design doc states it.
- That prior conversation context carries forward to a new task unless explicitly referenced.

## §10 Engineering Standards

### 10.1 General Principles

- **No magic numbers.** Named constants for all tuning values and thresholds.
- **No force unwraps or unchecked casts** unless at an unavoidable platform boundary with inline justification.
- **No `print()` debugging in committed code.** Use structured logging gated behind debug flags.
- **No stringly-typed APIs.** Use enums, constants, or typed identifiers.
- **No dead code.** If unused, delete it. No commented-out blocks.

### 10.2 File Discipline

- One primary concern per file. File name reflects its content.
- No empty, stub, or placeholder files.
- Prefer editing existing files over creating new ones. Every new file must be justified.
- No README, CONTRIBUTING, LICENSE, or CHANGELOG generation unless explicitly requested.

### 10.3 Architecture

- Clear separation between I/O boundaries (network, filesystem, user input) and core logic.
- Functions do one thing. Files contain one concept.
- Error handling at system boundaries. Internal code may trust its own invariants.
- No god files combining unrelated concerns.
- No circular dependencies between modules.
- No global mutable state.
- No over-engineering: no abstraction layers, factory patterns, or dependency injection frameworks beyond what the design doc specifies.

### 10.4 Security

**These rules are BLOCKING regardless of context.**

**Forbidden:**

- Hardcoded secrets, API keys, tokens, or credentials in source files.
- `.env` files committed to version control.
- Plaintext passwords in comments, logs, or test fixtures.
- `eval()` or equivalent dynamic code execution on untrusted input.
- Shell command construction from unvalidated strings.

**Mandatory:**

- Secrets loaded from environment variables or `.env` files (gitignored).
- All user-facing input validated at the boundary.
- HTTPS for all network calls. No HTTP fallbacks.
- Anthropic API key stored in environment, never in source.

## §11 Testing

### 11.1 Requirements

- Logic-bearing modules require at least one smoke test.
- Test names describe the scenario: `test_<unit>_<scenario>_<expected>` or the Swift equivalent using `@Test("description")`.
- No real network calls in tests. Mock or stub external services (especially the Anthropic API).
- No sleep-based synchronization. Use proper async test patterns (`XCTestExpectation`, Swift Testing `confirmation`).
- No hardcoded environment-specific paths in test fixtures.

### 11.2 Recommendations

- Tests live adjacent to the code they verify.
- Edge cases covered for any function handling user input or external data.
- Chapter state transitions tested for correctness and persistence.

## §15 Pull Request Governance

**Priority: BLOCKING. This section governs all PR creation and review behavior.**

### 15.1 Template Compliance

Every pull request MUST use the PR template (`.github/pull_request_template.md`). The following sections are mandatory and MUST NOT be left empty or deleted:

| Section                  | Requirement                                                                           |
| ------------------------ | ------------------------------------------------------------------------------------- |
| **Summary**              | Concise explanation of intent. Not a restatement of the diff.                         |
| **Design Doc Alignment** | Reference to the design doc section(s) implemented. Deviations declared explicitly.   |
| **Architectural Impact** | Affected modules, chapters, or services listed. Cross-module dependencies identified. |
| **Risk Surface**         | Failure modes and affected components described.                                      |
| **Testing**              | Device, OS version, scenarios verified, tests added. No exceptions.                   |
| **Protocol Zero**        | All assertions confirmed. `./protocol-zero.sh` must exit 0 on the branch.             |

**Dependency Changes** may state "None" if no dependencies were modified. **Evidence** is required when the change affects visual output, audio, or animation. **Reviewer Guidance** is required when the change introduces architectural uncertainty or tradeoffs.

**Agent Refusal Policy:** If a PR is missing mandatory sections, the agent MUST:

1. **HALT** — Do not create or submit the PR.
2. **REJECT** — State: "PR template sections are mandatory per §15.1."
3. **ASSIST** — Help the user complete the missing sections.

### 15.2 PR Title Standard

PR titles MUST follow the Conventional Commits format defined in §4.3:

```
type(scope): imperative description
```

**Constraints:**

- Under 70 characters total.
- Imperative mood. No trailing period.
- Scope required. Lowercase, hyphen-separated identifier for the affected area.
- Type must be one of: `feat`, `fix`, `refactor`, `test`, `chore`, `docs`.

**Valid examples:**

```
feat(chapter2): add haiku API response validation
fix(audio-service): resume BGM after system interruption
chore(ci): add protocol zero workflow
docs(design-doc): update state architecture guidance
refactor(state): migrate chapter progress to @Observable
```

**Rejected titles and reasons:**

| Title                                                | Violation                                        |
| ---------------------------------------------------- | ------------------------------------------------ |
| `fix stuff`                                          | No scope, vague description.                     |
| `updates`                                            | No type, no scope, no description.               |
| `WIP: chapter 1`                                     | WIP is not a valid type.                         |
| `Fixed the bug in audio`                             | Past tense, no scope delimiter.                  |
| `feat(chapter0,chapter1,state): overhaul everything` | Scope too broad. Split the PR.                   |
| `#456`                                               | Issue-only title. No context without navigation. |

**Agent Enforcement:** The agent MUST validate PR titles against this format before creating a PR. If a title does not conform, the agent MUST refuse and propose a corrected title. The agent MUST NOT create a PR with a non-conforming title.

### 15.3 Governance Authority in PR Context

**CLAUDE.md and the Design Document are the combined source of truth for all PR decisions.**

If any PR conflicts with either document, the agent MUST:

1. **HALT** — Do not approve, merge, or proceed.
2. **REJECT** — Cite the specific section of CLAUDE.md or the design document that is violated.
3. **WAIT** — Request clarification from the repository owner.

The agent MUST NOT approve or create PRs that:

- Introduce architectural patterns not sanctioned by the design document.
- Violate Protocol Zero (§2) in any file, comment, or commit message.
- Skip mandatory PR template sections.
- Contain non-conforming PR titles.
- Bundle unrelated changes into a single PR.
- Exceed reasonable scope without justification.

### 15.4 Squash Merge Governance

This repository uses squash merge (§4.2). In a squash merge workflow, the PR title becomes the commit message on the `main` branch. This makes the PR title the authoritative historical record.

**Implication:** Every constraint that applies to commit messages (§4.3) applies equally to PR titles. The PR title is not a summary — it is the production commit message.
