# Output Format

Strict template for the PHASES.md file generated by this skill.

## File Header

```markdown
# Implementation Roadmap — [Project Name]

**Source Document:** [path to design doc]
**Generated:** [date]
**Total Phases:** [count]
**Total Epics:** [count]

---
```

## Phase Summary Table

Immediately after the header. One row per phase.

```markdown
## Phase Summary

| Phase | Class          | Name   | Epics | Depends On | Entry Gate       | Exit Gate       |
| ----- | -------------- | ------ | ----- | ---------- | ---------------- | --------------- |
| 0     | Infrastructure | [name] | [n]   | None       | [entry criteria] | [exit criteria] |
| 1     | Infrastructure | [name] | [n]   | Phase 0    | [entry criteria] | [exit criteria] |
| 2     | Feature        | [name] | [n]   | Phase 0, 1 | [entry criteria] | [exit criteria] |
| ...   | ...            | ...    | ...   | ...        | ...              | ...             |
```

**Column definitions:**

- **Phase**: Sequential integer, zero-indexed, no gaps
- **Class**: Exactly one of `Infrastructure`, `Feature`, `Integration`
- **Name**: Short imperative descriptor (e.g., "Foundation and Scaffold", "Chapter 1 Runner Game")
- **Epics**: Integer count of epics in this phase
- **Depends On**: Comma-separated list of phase numbers that must complete first. "None" for Phase 0
- **Entry Gate**: Observable precondition (e.g., "Phase 0 exit criteria met, all Chapter 1 assets available")
- **Exit Gate**: Observable postcondition (e.g., "All Chapter 1 screens render, game plays to completion, tests pass")

## Dependency Graph (Text)

After the summary table, a text-based dependency visualization:

```markdown
## Dependency Graph

\`\`\`
Phase 0 ──► Phase 1 ──► Phase 2 ──► Phase 3
└──► Phase 4
Phase 3 ──► Phase 5
Phase 4 ──► Phase 5
Phase 5 ──► Phase 6
\`\`\`
```

Use ASCII box-drawing. Show all edges. Parallel paths on separate lines.

## Per-Phase Epic Breakdown

One section per phase. Each section contains:

```markdown
---

## Phase [N]: [Name]

**Class:** [Infrastructure | Feature | Integration]
**Depends on:** Phase [list]
**Design doc sections:** §[list of referenced sections]

### Entry Criteria

- [Observable condition 1]
- [Observable condition 2]

### Exit Criteria

- [Observable condition 1]
- [Observable condition 2]

### Epics

#### E[N].1: [Imperative title]

**Scope:** [What this epic covers]
**Deliverable:** [Single named output]
**Files touched:**
- `path/to/file1.swift` (create)
- `path/to/file2.swift` (create)

**Acceptance criteria:**
- [Binary verifiable condition]
- [Binary verifiable condition]

#### E[N].2: [Imperative title]

...
```

**Epic field definitions:**

- **Scope**: One sentence describing the boundaries of this epic
- **Deliverable**: The single tangible output (runnable screen, passing tests, working service)
- **Files touched**: Exhaustive list of files this epic creates or modifies, with action (create/modify)
- **Acceptance criteria**: 2-5 binary conditions that determine if the epic is done

## File Footer

```markdown
---

## Execution Notes

- Phases execute sequentially. Do not begin Phase N+1 until Phase N exit criteria are met.
- Epics within a phase may execute in parallel if their file sets are disjoint.
- If an epic's acceptance criteria cannot be met, stop and reassess before continuing.
- This roadmap is deterministic. Follow it top-to-bottom.
```

## Validation Checklist

Before finalizing PHASES.md, verify:

- [ ] Every phase has a unique sequential number starting from 0
- [ ] Every phase has exactly one class label
- [ ] Every phase has explicit entry and exit criteria
- [ ] Every phase depends on at least one prior phase (except Phase 0)
- [ ] Every epic has a named deliverable
- [ ] Every epic lists files touched
- [ ] No two epics in the same phase share a file
- [ ] The dependency graph is acyclic
- [ ] Total epic count matches the sum in the summary table
- [ ] All design doc sections are referenced by at least one phase
